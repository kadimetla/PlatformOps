PlatformOps Agent

Harness Architecture Deep Dive

From a chat message to reviewed, approved, provisioned cloud infrastructure — the flow, the layers, and what's built vs. designed.

Source of truth for everything in this deck: docs/HARNESS_DESIGN.md in this repo. This deck is a visual companion to that doc, not a replacement for it.

Why a harness, not just an agent script

Today

agents/orchestrator.py is invoked directly: one process, one session, one requester, no identity across requests. Fine for a hackathon demo.

Real usage needs

  • Multiple teams, multiple concurrent requests
  • Requests from wherever people already work — Slack, a ticket, a CLI
  • A human able to see and approve a pending action before it runs
  • Isolation so one org/team's data never leaks into another's
Wrong tool for this job: a bigger prompt. Right tool: a harness that owns sessions, routing, and review as first-class concepts.

Borrow OpenClaw's shape, build our own Gateway

OpenClaw solves a structurally similar problem for coding agents. We use it as a design reference, not as infrastructure we run on — see the decision trail below.

Considered: CLI Backend Plugin declined

Wraps existing CLI tools only. Would need an adapter-shim CLI in front of our ADK graph, losing structured session/tool-call fidelity.

Considered: Register as MCP tool on real OpenClaw declined for now

Genuinely low-cost via OpenClaw's documented mcp.servers config. Declined because it ties us to OpenClaw's roadmap and a Control UI built for chat, not infra approval.

Chosen: Custom PlatformOps Gateway decided

Full control over the approval/dispatch model. OpenClaw is a pattern library (bindings, isolation, config validation, Control UI, plugin policy) — not our runtime.

Earlier design draft recommended a "plugin-harness" running inside OpenClaw's runtime — corrected after reading concepts/agent, which states OpenClaw's embedded runtime is single and non-pluggable.

Multi-tenancy: Org → Business Unit → isolation unit

OpenClaw's isolation unit (agent_id) is flat — no native org/BU nesting. We layer that hierarchy in our own config.

Org: acme  (exists only in our config layer)

BU: payments

agent_id: acme-payments
workspace bundle: AWS creds, allowed resource types, cost ceiling, model overrides

BU: platform

agent_id: acme-platform
workspace bundle: separate AWS creds, separate allow-list, separate ceiling
Hard rule from OpenClaw's own docs: "never reuse agentDir across agents" — onboarding must always mint a fresh agent_id per BU, never reuse one.

Agent layer — what's actually built today

platformops_orchestrator
  ├── provisioning_agent (router)
  │    ├── cdk_provisioning_agent
  │    │    ├── aws-iac-mcp-server (docs / validation, read-only)
  │    │    └── ccapi-mcp-server (AWS mutation)
  │    └── terraform_provisioning_agent
  │        └── HashiCorp Terraform MCP Server
  └── security_agent (no tools — approve/reject only)

What's solid built

Routing, execution, and review are separate roles with different model tiers. security_agent has zero mutating tools.

What's still prompt-level gap

Mutating MCP tools are attached directly to provisioning agents. The approval rule is an instruction, not a runtime guard — that's the enforcement gap the Gateway closes.

End-to-end request flow

1
Request arrives via Slack, CLI, webhook, or ticketing.
2
Gateway normalizes it into a typed envelope and resolves org, BU, agent_id, workspace bundle, model overrides, and policy bundle via deterministic bindings.
3
Workflow runtime starts a durable lifecycle and runs deterministic checks (spec/check_compliance.py) before any LLM agent runs.
4
ADK agent graph drafts a plan and Vibe Diff, using read-only validation tools first.
5
Security agent reviews the plan; approval is recorded as structured data, not just chat text.
6
If policy requires a human, the workflow pauses and the Control UI shows the pending approval.
7
Any cloud mutation becomes a ToolIntent; the policy dispatcher checks plan hash, approval, workspace policy, cost, region, resource type, credential scope.
8
Only an allowed ToolIntent reaches CCAPI or Terraform apply. Every denial, dispatch, result, and verification is written to audit.

The enforcement gap this closes: ToolIntent

Today, mutating MCP tools are attached directly to provisioning agents and the approval rule is a prompt instruction. The Gateway replaces that with a hard runtime boundary:

execution agent → proposed ToolIntent → Gateway policy dispatcher → MCP / cloud tool

ToolIntent carries

  • plan_id + exact plan version hash
  • org, BU, requester, source channel
  • target provider/account/region
  • tool name + normalized action
  • resource type, resource id, operation
  • expected diff + estimated monthly cost
  • approval requirement + approval record id

Dispatcher rule

Deny by default. Only dispatches when the intent matches an approved plan and the workspace bundle permits the resource type, account, region, and operation.

This is the difference between a demo agent graph and a production harness.

What's built vs. designed, right now

Built built

  • Agent layer: orchestrator, router, CDK/Terraform sub-agents, security agent
  • Model config file + loader (config/models.yaml, agents/model_config.py)

Designed, not built designed

  • Gateway process, channel adapters, session/routing layer
  • Control UI / human-in-the-loop approval queue
  • Org → BU → agent_id multi-tenancy model + org registry
  • True model-agnosticism beyond Gemini
  • RequestEnvelope / WorkspaceBundle / PlanRecord / ApprovalRecord / ToolIntent schemas
Decided direction: custom PlatformOps Gateway (own runtime, not OpenClaw's). See docs/HARNESS_DESIGN.md for the full decision trail.
